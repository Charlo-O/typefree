# 2026-03-08 更新说明

本文记录本轮围绕悬浮窗、热键、剪贴板、AssemblyAI 接入、提示词、稳定性和发布流程的集中调整，便于后续继续迭代和排查。

## 1. 悬浮窗与窗口行为

### 1.1 启动行为调整

- 应用启动时，主悬浮窗不再直接显示在桌面上。
- 听写相关界面改为在需要时再显示，减少开机或重启后的桌面干扰。

### 1.2 悬浮窗位置与尺寸

- 主悬浮窗在显示时会被缩小，并放置在屏幕中下区域，更贴近“随手可见但不遮挡”的使用方式。
- 主悬浮窗按钮和弹出菜单都做了收缩，减少视觉占用。

### 1.3 剪贴板窗口独立化

- 剪贴板内容不再复用控制面板窗口。
- 双击剪贴板快捷键后，会弹出一个独立的剪贴板窗口，只显示剪贴板相关内容。

## 2. 热键与交互逻辑

### 2.1 听写快捷键与剪贴板快捷键分离

- 听写快捷键和剪贴板快捷键已经拆分为两个独立设置项。
- 剪贴板快捷键显示在设置中的激活方式下面，避免与听写激活方式混淆。

### 2.2 剪贴板快捷键行为

- 剪贴板快捷键是单独的单键设置。
- 触发方式为“快速双击同一个按键”。
- 该按键为全局热键，因此如果把字母、数字等常用键设置为剪贴板热键，运行中可能会占用该键的正常输入能力。

### 2.3 热键稳定性修复

- 控制面板窗口不再在启动时重复注册全局热键。
- 后端热键注册增加了串行化保护，减少 Windows 上因并发注册/反注册带来的闪退风险。

## 3. 语音转文字链路

### 3.1 AssemblyAI 接入

- AssemblyAI 已加入语音转文字提供方，并放在提供方列表前面。
- 设置页在选择 AssemblyAI 时会显示专用的“转录提示词”输入框。
- 语音转文字页面不再承载“文字修饰提示词”，该类提示词仍由原有 Prompt Studio 统一管理。

### 3.2 中文与模型兼容处理

- 中文场景下，为避免 `universal-3-pro` 直接拒绝 `zh`，请求链路补充了 `speech_models` 回退处理。
- 当前实现会在支持场景优先走更强模型，在不支持时回落到兼容模型。

### 3.3 超时控制

- 录音结束后的处理链路增加了 60 秒超时保护。
- 如果转录、等待结果或后处理总耗时超过 60 秒，流程会被主动中止，避免长时间无响应。

### 3.4 日志与诊断

- 前端和后端都补充了更细的转录日志。
- 日志会记录音频大小、模型、提示词是否带入、请求阶段以及性能分段耗时，便于定位是“上传慢”“服务端转录慢”还是“推理慢”。

## 4. AI 文本整理与提示词

### 4.1 默认提示词改为中文

- 默认系统提示词已改为中文，降低中文用户直接修改提示词时的理解成本。
- Prompt Studio 现在会统一读取当前生效的提示词模板，显示与运行时行为保持一致。

### 4.2 默认策略从“回答问题”改为“整理原文”

- 默认行为改为：只整理用户说出的原话，不主动回答问题。
- 当用户口述“为什么……”“多久……”“怎么……”“什么原因……”等问题句时，系统会把它们当作听写内容清理后输出，而不是生成答案。
- 只有在识别到明确的编辑指令时，才执行改写、润色、总结、翻译、列表化等额外处理。

### 4.3 旧默认提示词迁移

- 如果本地保存的是旧版英文默认提示词，系统会自动回退到新的中文默认提示词。
- 如果用户曾经手动编辑并保存过自定义提示词，则不会被这次迁移覆盖。

## 5. Reasoning 与性能观察

### 5.1 当前性能结论

- 语音到输出的总耗时主要由两段决定：
  - 语音转文字服务端处理时间
  - 转录完成后的 Reasoning 模型后处理时间

### 5.2 已确认的性能特征

- 在使用 `deepseek-reasoner` 时，Reasoning 常常比转录本身更慢。
- 在切换到 `deepseek-chat` 后，Reasoning 耗时会明显下降。
- 当转录服务返回速度波动较大时，即使 Reasoning 很快，整条链路仍然会被语音转文字阶段拖慢。

### 5.3 当前建议

- 若追求更快的听写结果，可优先关闭 AI 文本增强，或使用更轻量的非推理模型。
- 若需要严格控制总耗时，可继续保留 60 秒超时保护，并结合日志观察瓶颈位置。

## 7. CI/CD 发布流程

### 7.1 发布触发方式

- 支持推送 `vX.Y.Z` 标签触发发布。
- 也支持在 GitHub Actions 中手动输入版本号触发发布。

### 7.2 发布产物

- Windows、macOS 和 Linux 安装包会继续作为 Release 资产上传。
- 此外，工作流会在全部打包成功后，额外把当前版本的完整源码打成压缩包并上传到 GitHub Release。

### 7.3 这样做的目的

- 安装包和源码版本一一对应，便于回溯、归档和外部审计。
- 对外发布时，除了平台安装包，还能直接下载到当前标签对应的完整源码快照。
